## 1. 프로젝트 개요

**Proposal Agent**는 RFP(제안요청서) 문서를 입력받아 Impact-8 구조의 PPTX 제안서를 자동 생성하는 AI 에이전트 시스템이다.

### 핵심 가치
- **RFP 파싱**: PDF/DOCX/TXT/PPTX 등 다양한 형식의 RFP 문서를 파싱
- **RFP Chunking**: 섹션 우선순위 기반 청킹으로 평가기준·요구사항을 완전 분석 (25,000자 → 40,000자)
- **LLM 기반 분석·생성**: Claude / Gemini / Groq / Ollama 중 하나를 선택해 전략적 RFP 분석 및 제안서 콘텐츠 생성
- **Cross-Phase Context**: Phase 간 핵심 결론 전달로 내러티브 일관성 확보
- **Industry Stats DB**: 업종별 검증 통계 주입으로 수치 구체성 향상
- **Slide Quality Scoring**: 규칙 기반 자동 채점으로 품질 미달 슬라이드 조기 탐지
- **Phase Checkpoint**: Phase 완료마다 자동 저장으로 API 실패 복구 가능
- **수동 모드(Manual Mode)**: `generate --manual` / `continue` / `status` — LLM API 없이 파일 기반으로 9단계 진행 (과금·토큰 한도 회피)
- **자동 PPTX 생성**: Modern 스타일 PPTX로 렌더링 (폰트 폴백 포함)

### 기술 스택

| 분류      | 라이브러리                          | 역할                         |
| --------- | ----------------------------------- | ---------------------------- |
| CLI       | typer, rich                         | 커맨드라인 인터페이스 및 콘솔 출력 |
| AI/LLM    | anthropic, google-genai, groq       | LLM API 호출                 |
| 문서 파싱  | pypdf, pdfplumber, python-docx      | PDF/DOCX 파싱                |
| PPTX 생성  | python-pptx                         | 슬라이드 생성                 |
| 데이터 검증 | pydantic v2                         | 스키마 정의 및 타입 검증      |
| 로깅      | loguru                              | 구조화 로깅 ({} 스타일)      |
| 설정      | python-dotenv                       | 환경변수 로드                |

## 3. 전체 아키텍처 및 데이터 흐름

```
[사용자 CLI 입력]
    │  python main.py generate input/rfp.pdf -n "..." -c "..." -t marketing_pr
    ▼
[main.py — Typer CLI]
    │  rfp_path, project_name, client_name, proposal_type, company_data 파싱
    ▼
[ProposalOrchestrator.execute()]
    │
    ├─► [파서 선택: get_parser_for_path(rfp_path)]
    │       PDF → PDFParser | DOCX → DOCXParser | TXT → TXTParser | PPTX → PPTXParser
    │       반환: {"raw_text": str, "tables": [...], "sections": [...]}
    │
    ├─► [RFPAnalyzer.execute(parsed_data)]
    │       ├─ RFPChunker.build_analysis_context() — 섹션 우선순위 청킹 (40,000자)
    │       ├─ LLM 호출 (RFP 분석)
    │       └─ 반환: RFPAnalysis (project_name, pain_points, win_theme_candidates, evaluation_strategy 등)
    │
    ├─► [ContentGenerator.execute(rfp_analysis, company_data)]
    │       Phase 0: HOOK
    │       Phase 1~7: 각 Phase 생성 루프
    │           ├─ Cross-Phase Context: 이전 Phase 결론 주입
    │           ├─ Industry Stats DB: 업종별 통계 주입
    │           ├─ Negative Prompts: 플레이스홀더·막연한 표현 금지
    │           ├─ LLM 호출 (_call_llm_and_extract_json)
    │           ├─ Slide Quality Scoring: 자동 채점 (경고 로그)
    │           └─ Phase Checkpoint: output/_checkpoints/ 저장
    │       반환: ProposalContent
    │
    ▼
[PPTXOrchestrator.execute(content, output_path)]
    │   Phase별 슬라이드 유형 → 생성 메서드 매핑
    │   TemplateManager: 폰트 폴백, 디자인 시스템
    │   PPTXGenerator / ChartGenerator / DiagramGenerator 호출
    ▼
[output/프로젝트명_YYYYMMDDHHmmssfff.pptx]
```

## 5. 진입점: main.py

### CLI 명령어 (Typer 앱)

| 명령              | 필수 인자/옵션 | 생략 시 기본값 요약 | 설명                                   |
| ----------------- | -------------- | -------------------- | -------------------------------------- |
| `generate`        | RFP_PATH (단, `-r` 사용 시 run_metadata 있으면 생략 가능) | `-n`·`-c` RFP 추출, `-d` company_profile.json, `-o` output, `--manual` API 모드 | RFP → PPTX 전체 파이프라인. `--manual` 시 수동 모드 |
| `continue`        | 없음           | `--manual-dir` → 최신 run | 수동 모드: 현재 단계 응답 처리 후 다음 단계      |
| `manual-run`      | `--site` (gemini\|chatgpt) | RFP_PATH(없으면 최신 run), `--manual-dir` | **1~9단계 전체 자동**: 로그인만 사람이 함, request→전송→response 저장→continue 반복으로 PPTX까지 생성. Step 1 로그인 후 같은 터미널 Enter로 진행 |
| `status`          | 없음           | `--manual-dir` → 최신 run | 수동 모드 진행 상태 확인                     |
| `analyze`         | RFP_PATH       | —                    | RFP 분석만 수행                           |
| `setup-company`  | 없음           | `-o` company_data/company_profile.json | 회사 프로필 대화형 설정                        |
| `types`           | 없음           | —                    | 제안서 유형 목록                            |
| `templates`       | 없음           | —                    | PPTX 템플릿 목록                           |
| `info`            | 없음           | —                    | Impact-8 Framework 설명                   |
| `help`            | 없음           | —                    | 명령별 상세 예시(필수/기본값 포함)              |

상세한 필수·선택(생략 시 기본값)은 `python main.py help` 또는 [2.INSTALL_AND_USAGE.md](2.INSTALL_AND_USAGE.md) § 명령별 필수·선택 요약 참고.

## 7. 에이전트 레이어

### src/agents/base_agent.py — BaseAgent

모든 에이전트의 기반 클래스.

**핵심 메서드:**

|     | 메서드                                                         | 설명                                        |     |
| --- | ----------------------------------------------------------- | ----------------------------------------- | --- |
|     | `_call_llm(system, user, max_tokens)`                       | LLM API 호출 (재시도·지수 백오프)                   |     |
|     | `_call_llm_and_extract_json(system, user, ..., retry_hint)` | JSON 추출 실패 시 `LLM_JSON_RETRY_COUNT`만큼 재시도 |     |
|     | `_extract_json_from_response(text)`                         | 응답에서 JSON 블록 추출                           |     |
|     | `_normalize_json_keys(data, aliases)`                       | camelCase 등 키 → snake_case 정규화            |     |
|     | `_truncate_text(text, max_chars)`                           | 텍스트 길이 제한                                 |     |

## 17. LLM 호출 메커니즘 상세

### BaseAgent._call_llm_and_extract_json

```
1. LLM 호출 (_call_llm)
2. _extract_json_from_response: ```json ... ``` 블록 추출
3. JSON 파싱 시도
4. 실패 시 retry_hint와 함께 LLM 재호출 (LLM_JSON_RETRY_COUNT번까지)
5. 최종 실패 시 ValueError(JSON_PARSE_FAILED_MESSAGE)
```

### 재시도 전략

- **API 오류 (429, 5xx)**: `LLM_RETRY_COUNT`번, 지수 백오프 (`LLM_RETRY_BASE_DELAY`)
- **JSON 파싱 실패**: `LLM_JSON_RETRY_COUNT`번, 동일 프롬프트에 JSON 재요청 hint 추가
- **호출 간 대기**: `LLM_DELAY_SECONDS`(기본 8초)로 무료 한도 429 방지

### Groq 특이 처리

- `GROQ_MAX_USER_MESSAGE_CHARS`(0=제한없음): 413 Request Too Large 방지
- `GROQ_MAX_REQUEST_TOKENS`(기본 5000): 요청 전체 토큰 상한 (한글 2자/토큰 보수적 추정)
- 토큰 초과 예상 시 user 메시지 마지막 부분부터 잘라서 전송

## 21. 핵심 설계 패턴

### 1. 계층화된 품질 향상

```
RFP 입력 품질       → RFPChunker (섹션 우선순위 청킹)
프롬프트 품질        → Industry Stats + Negative Prompts
생성 일관성          → Cross-Phase Context
사후 검증            → SlideQualityScorer (규칙 기반)
안정성               → Phase Checkpoint
```

### 2. 환경변수 기반 기능 토글

모든 고도화 기능은 `.env` 환경변수로 on/off 가능. 기본값은 모두 사용자에게 유리한 방향(`true`). `ENABLE_SELF_REFINEMENT`만 토큰 비용 고려로 `false` 기본.

### 3. 싱글톤 Settings

`get_settings()`로 전역 접근. 최초 1회 생성 후 캐시. Pydantic validator로 범위 검증.

### 4. JSON 파싱 내성

모든 LLM 응답은 `_call_llm_and_extract_json()`로 처리 — `retry_hint`를 포함한 재시도로 JSON 파싱 실패율 최소화. 최종 실패 시 표준 에러 메시지로 명확한 오류 전달.

### 5. 폰트 폴백 (TemplateManager)

템플릿 PPTX의 폰트를 추출 후 `_safe_font_name()`으로 시스템 설치 여부 확인. 미설치 폰트(Pretendard 등)는 `맑은 고딕`으로 자동 대체하여 렌더링 오류 방지.

### 6. 회사 프로필 활용 흐름

```
python main.py setup-company
  → company_data/company_profile.json 생성

python main.py generate ...
  → ProposalOrchestrator: company_profile.json 로드
  → ContentGenerator Phase 6: company_data를 프롬프트에 주입
  → WHY US 슬라이드에 실제 역량·실적 반영
```

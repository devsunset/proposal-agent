# 설치 및 사용 가이드

설치·환경 설정·실행·트러블슈팅을 한 문서에서 다룹니다.  
LLM은 `.env`의 `LLM_PROVIDER`로 선택 (claude | gemini | groq | ollama). 로컬 LLM(Ollama)은 아래 「2-5. Ollama 설치」 참고.

---

## 1. 사전 요구사항

| 항목       | 요구 사항                                                                 |
| ---------- | ------------------------------------------------------------------------- |
| **Python** | 3.10 이상                                                                 |
| **pip**    | 최신 버전 권장                                                            |
| **LLM**    | Claude / Gemini / Groq API 키 또는 **Ollama**(로컬, API 키 불필요). `.env`의 `LLM_PROVIDER`에 맞춤 |

> **세 가지 방식으로 사용할 수 있습니다**
> - **① AI 코드 어시스턴트 방식** — Cursor 등 AI 도구에게 말로 시키면 RFP 분석부터 PPTX 생성까지 자동 처리
> - **② CLI(API) 방식** — `python main.py generate` 명령어로 실행. `.env`의 LLM_PROVIDER와 API 키 필요
> - **③ 수동 모드** — `python main.py generate <rfp> --manual` 후, `manual_req_res/run_*/N_step_{Phase명}_request.txt`의 프롬프트를 Gemini 등에 붙여넣고 응답을 해당 `*_response.txt`에 저장한 뒤 `python main.py continue` 반복. **브라우저 자동화**: `python main.py manual-step --site gemini` 또는 `--site chatgpt` 로 현재 단계를 자동 전송·저장 가능(Playwright 설치 필요). API 비용·토큰 한도 없음

### 2-5. Ollama 설치 (로컬 LLM 사용 시)

`LLM_PROVIDER=ollama` 로 사용하면 **API 키 없이** 로컬 LLM으로 동작합니다. 비용·요청 한도 제한이 없습니다.

#### 왜 Ollama를 쓰나요?

| 항목       | 설명                                                                 |
| ---------- | -------------------------------------------------------------------- |
| **API 키** | 불필요. 로컬 `http://localhost:11434` 로만 통신.                     |
| **비용·한도** | 로컬 실행이라 **비용 없음**, **요청 한도 없음**.                   |
| **품질·속도** | 모델(예: llama3.2, qwen2.5)과 PC 사양(CPU/GPU, RAM)에 따라 상이. GPU 있으면 더 빠르고 품질 좋음. |

#### Ollama 설치

**Windows**

1. [ollama.com/download](https://ollama.com/download)에서 **OllamaSetup.exe** 다운로드.
2. OllamaSetup.exe 실행 → UAC 승인 → 설치 완료. (사용자 홈 기준 설치, 관리자 권한 불필요.)
3. 설치 후 Ollama가 백그라운드에서 실행되며 **http://localhost:11434** 에 API 서버가 떠 있습니다.
4. 터미널에서 확인: `ollama --version`

**macOS**

1. [ollama.com/download](https://ollama.com/download)에서 macOS용 설치 파일 다운로드.
2. 다운로드한 이미지를 **Applications** 로 드래그하거나 설치 절차 진행.
3. 앱 실행 후 **메뉴바에 Ollama 아이콘**이 보이면 서버가 동작 중입니다.

**Linux**

1. 설치: `curl -fsSL https://ollama.com/install.sh | sh`
2. 서비스 기동 (일부 배포판은 설치 시 자동 등록): `ollama serve`

#### 모델 내려받기 및 확인

```bash
ollama pull llama3.2
ollama list   # 여기 나오는 이름을 .env 의 OLLAMA_MODEL 에 그대로 넣음
```

- 다른 모델 예: `ollama pull qwen2.5:7b` (다국어·한글에 유리), `ollama pull llama3.1:8b`

| 모델         | 명령                     | 비고             |
| ------------ | ------------------------ | ---------------- |
| llama3.2     | `ollama pull llama3.2`   | 기본 추천, 가벼움 |
| qwen2.5:7b   | `ollama pull qwen2.5:7b`  | 한글·다국어       |
| llama3.1:8b  | `ollama pull llama3.1:8b` | 품질·용량 균형    |

#### .env 설정 (Ollama 사용 시)

```env
LLM_PROVIDER=ollama
OLLAMA_BASE_URL=http://localhost:11434/v1
OLLAMA_MODEL=llama3.2:latest
```

- **OLLAMA_BASE_URL**: 기본값이 `http://localhost:11434/v1` 이므로 생략 가능.
- **OLLAMA_MODEL**: `ollama list` 에 나오는 이름과 맞추기 (예: `llama3.2:latest`, `qwen2.5:7b`).

#### 실행 전 확인

1. **Ollama가 켜져 있는지**  
   Windows/macOS: Ollama 앱이 실행 중인지 확인. Linux: `ollama serve` 또는 해당 서비스가 떠 있는지 확인.
2. **API 동작 확인** (선택): `curl http://localhost:11434/api/tags` — 모델 목록이 JSON으로 나오면 정상.

#### 제안서 생성·분석 실행 (Ollama 사용 시)

```bash
pip install -r requirements.txt
python main.py generate input/sample.txt   # 제안서 생성 (RFP 경로는 실제 파일로 변경)
python main.py analyze input/sample.txt    # RFP 분석만
```

콘솔에 **Ollama (로컬)** 로 표시되면 Ollama 연동으로 동작하는 것입니다.

#### Ollama 문제 해결

| 현상                            | 확인·조치                                                                 |
| ------------------------------- | ------------------------------------------------------------------------- |
| **Connection refused / 연결 실패** | Ollama 실행 여부 확인. Windows/macOS: 앱 실행, Linux: `ollama serve`      |
| **model not found**             | `OLLAMA_MODEL`을 `ollama list` 이름과 맞추기. `ollama pull <모델명>` 후 `.env` 설정 |
| **응답이 느림**                  | 모델 크기·PC 사양(GPU) 영향. 작은 모델(예: llama3.2) 또는 GPU 드라이버 확인 |
| **JSON 파싱 실패·형식 깨짐**     | `LLM_TEMPERATURE=0.3`으로 낮추거나, qwen2.5:7b·llama3.1:8b 등 큰 모델 사용   |

**요약:** [ollama.com/download](https://ollama.com/download)에서 OS에 맞게 설치 → `ollama pull llama3.2` (또는 원하는 모델) → `.env`에 `LLM_PROVIDER=ollama`, `OLLAMA_MODEL=llama3.2:latest` 설정 → `python main.py generate input/...` 또는 `python main.py analyze input/...` 실행.

**대안: 수동 모드** — LLM API를 전혀 쓰지 않으려면 `python main.py generate <rfp> --manual` 후, `manual_req_res/run_*/N_step_{Phase명}_request.txt`의 프롬프트를 Google Gemini(https://gemini.google.com/) 또는 ChatGPT(https://chat.openai.com/) 등에 붙여넣고 응답을 해당 `*_response.txt`에 저장한 뒤 `python main.py continue`를 반복하면 됩니다. `python main.py manual-step --site gemini`(또는 `--site chatgpt`)로 브라우저 자동화도 가능합니다. 본문 「수동 모드」 절 참고.

## 3. 사용법

### 방법 ① AI 코드 어시스턴트로 제안서 생성 (권장)

```
사용자: "제안요청서 폴더의 테스트 01 파일을 분석한 후 제안서를 제작해줘"

AI가 자동으로 수행:
  1. RFP Chunking으로 평가기준·요구사항 우선 파싱 (최대 40,000자)
  2. Impact-8 구조로 콘텐츠 기획 + Win Theme 3개 도출
  3. Cross-Phase Context + Industry Stats 주입으로 일관성 있는 제안서 생성
  4. PPTX 렌더링 (폰트 폴백 적용)
  5. Slide Quality Score 확인 → 품질 미달 슬라이드 수정
```

### 방법 ② CLI 명령어

#### (최초 1회) 회사 프로필 설정

```bash
python main.py setup-company
```

회사 정보를 대화형으로 입력하면 `company_data/company_profile.json`이 생성됩니다.
이후 모든 제안서의 **Phase 6(WHY US)**에 실제 역량·실적이 자동 반영됩니다.

#### 제안서 생성

```bash
python main.py generate input/rfp.pdf -n "프로젝트명" -c "발주처명"
```

| 옵션                   | 설명                     | 필수 | 생략 시 기본값 |
| ---------------------- | ------------------------ | ---- | ---------------- |
| `input/rfp.pdf`        | RFP 파일 경로 (PDF/DOCX/TXT/PPTX). `--resume-checkpoint` 사용 시 run_metadata.json 없을 때만 필요 | O (재개 시 생략 가능) | — |
| `-n` / `--name`        | 프로젝트명 (미입력 시 RFP에서 추출) | X    | RFP에서 추출 |
| `-c` / `--client`      | 발주처명 (미입력 시 RFP에서 추출) | X    | RFP에서 추출 |
| `-t` / `--type`        | 제안서 유형 (marketing_pr, event, it_system, public, consulting, general) | X    | 자동 판별 |
| `-d` / `--company`    | 회사 정보 JSON 경로      | X    | company_data/company_profile.json |
| `-o` / `--output`     | 출력 디렉토리            | X    | output/ |
| `-T` / `--template`   | PPTX 템플릿 파일명 (확장자 제외) | X    | 기본 디자인(미지정) |
| `--save-json`          | 콘텐츠 JSON 저장         | X    | 미저장 |
| `--manual` / `-m`     | 수동 모드 (LLM API 없이 파일 기반) | X    | API 모드 |
| `--resume-checkpoint` / `-r` | 체크포인트 재개 (`_checkpoints/run_YYYYMMDD_HHMMSS` 또는 `run_...`). 지정 시 RFP 경로 생략 가능(run_metadata.json 있을 때) | X    | 없음(일반 실행) |

※ 모든 명령에서 **필수**가 아닌 인자·옵션은 생략 가능하며, 생략 시 위 **생략 시 기본값**이 적용됩니다.

#### 수동 모드 (LLM API·과금 없이 진행)

API 호출 없이, 프롬프트를 사람이 Google Gemini 또는 ChatGPT 등에 붙여넣고 응답을 파일로 저장한 뒤 단계별 진행합니다.

```bash
# 1. 수동 모드 시작 → manual_req_res/run_YYYYMMDD_HHMMSS/1_step_RFPAnalyzer_request.txt 생성
python main.py generate input/rfp.pdf --manual -n "프로젝트명" -c "발주처"

# 2. [방법 A] 수동: request 파일의 [시스템 프롬프트]·[사용자 메시지]를 Gemini 또는 ChatGPT에 입력 → 응답을 해당 *_response.txt 에 붙여넣기
# 2. [방법 B] 브라우저 자동화 (권장): Playwright 설치 후 한 단계씩 자동 전송·저장
playwright install chromium
python main.py manual-step --site gemini    # 또는 --site chatgpt
# → 브라우저에서 로그인한 뒤 **같은 터미널에서 Enter** 입력 시 질의 전송 → 응답이 끝날 때까지 대기 후 저장·다음 단계 진행
# 실행 시 콘솔에 [manual-step] (1/9)~(9/9) 단계별 진행 로그가 출력됩니다.

# **1~9단계 전체 자동 실행** (한 번에 request·response 반복 → PPTX까지)
python main.py manual-run --site gemini    # 또는 --site chatgpt (RFP 생략 시 최신 run)
python main.py manual-run --site gemini input/rfp.pdf   # RFP 지정 시 해당 run 사용(없으면 새 run 생성)
# → Step 1에서 한 번 로그인(같은 터미널 Enter) 후 Step 2~9는 자동으로 request 전송·response 저장·다음 단계 요청 생성이 반복됩니다.

# 3. 다음 단계 진행 (Step 2~9 반복)
python main.py continue

# 진행 상태 확인
python main.py status
```

- **총 9단계**: Step 1 RFP 분석 → Step 2 Phase 0 (HOOK) → … → Step 9 Phase 7 (INVESTMENT) → PPTX 생성
- **실행 시점별 폴더**: 매번 `generate --manual` 시 `manual_req_res/run_YYYYMMDD_HHMMSS/` 가 새로 생성되어 덮어쓰기 없음. `continue`/`status` 는 기본값으로 `latest_run.txt` 에 기록된 최신 run 폴더 사용.
- `continue`, `status` 에서 `--manual-dir manual_req_res/run_YYYYMMDD_HHMMSS` 로 특정 run 지정 가능

**manual-step 로그인이 안 될 때**

| 원인 | 설명 | 해결 |
|------|------|------|
| **매번 새 세션** | 예전에는 Playwright가 매번 새(시크릿 같은) 프로필로 열어 로그인·쿠키가 저장되지 않음 | 지금은 기본으로 **영구 프로필**(`manual_req_res/.browser_profile_gemini` 또는 `.browser_profile_chatgpt`) 사용 → 한 번 로그인하면 다음 실행부터 유지됨 |
| **자동화 탐지** | Google/OpenAI가 자동화 브라우저를 감지해 차단·추가 인증을 요구할 수 있음 | 기본으로 **시스템 Chrome**(`--browser-channel chrome`) 사용. Chrome 미설치 시 `playwright install chromium` 후 `--browser-channel chromium` 지정 |
| **Chrome 미설치** | 기본값이 Chrome이라 Chrome이 없으면 실행 실패 | Chrome 설치하거나, `python main.py manual-step --site gemini --browser-channel chromium` 처럼 `chromium` 지정 |

옵션 요약: `--browser-channel chrome`(기본) | `msedge` | `chromium`, `--user-data-dir <경로>`(프로필 폴더), `--no-persistent-profile`(영구 프로필 비사용, 디버깅용).

**수동 모드 상세 (폴더 구조·단계 매핑·구현 참고)**

- **개요**: LLM API 대신 단계별 프롬프트를 파일로 저장하고, 사용자가 Gemini 등에서 질의·응답을 수작업 진행. 과금·속도 제한 없이 무료 Gemini 활용 가능.

**폴더/파일 구조**

파일명 규칙: `{Step번호}_step_{Phase명}_request.txt` / `_response.txt`  
(Step 1: `RFPAnalyzer`, Step 2~9: `HOOK`, `SUMMARY`, `INSIGHT`, `CONCEPT`, `ACTION_PLAN`, `MANAGEMENT`, `WHY_US`, `INVESTMENT`)

```
manual_req_res/            # 수동 모드 기준 폴더 (실행 시점별 run_* 하위 생성)
├── latest_run.txt         # 최신 run 폴더명 (continue/status 기본 해석용)
├── run_YYYYMMDD_HHMMSS/   # 실행 시점별 폴더 (예: run_20250305_093000)
│   ├── state.json
│   ├── 1_step_RFPAnalyzer_request.txt / 1_step_RFPAnalyzer_response.txt
├── 2_step_HOOK_request.txt / 2_step_HOOK_response.txt
├── ... (3~9 동일 패턴)
└── 9_step_INVESTMENT_request.txt / 9_step_INVESTMENT_response.txt
```

**단계별 매핑**

| Step | 내용                    | 요청 파일                         | 응답 파일                          |
| ---- | ----------------------- | --------------------------------- | ---------------------------------- |
| 1    | RFP 분석 (RFPAnalyzer)  | `1_step_RFPAnalyzer_request.txt`  | `1_step_RFPAnalyzer_response.txt` |
| 2    | Phase 0: HOOK (티저)    | `2_step_HOOK_request.txt`        | `2_step_HOOK_response.txt`        |
| 3    | Phase 1: SUMMARY        | `3_step_SUMMARY_request.txt`     | `3_step_SUMMARY_response.txt`     |
| 4    | Phase 2: INSIGHT        | `4_step_INSIGHT_request.txt`     | `4_step_INSIGHT_response.txt`     |
| 5    | Phase 3: CONCEPT        | `5_step_CONCEPT_request.txt`     | `5_step_CONCEPT_response.txt`     |
| 6    | Phase 4: ACTION PLAN    | `6_step_ACTION_PLAN_request.txt` | `6_step_ACTION_PLAN_response.txt` |
| 7    | Phase 5: MANAGEMENT     | `7_step_MANAGEMENT_request.txt`  | `7_step_MANAGEMENT_response.txt`  |
| 8    | Phase 6: WHY US         | `8_step_WHY_US_request.txt`      | `8_step_WHY_US_response.txt`      |
| 9    | Phase 7: INVESTMENT     | `9_step_INVESTMENT_request.txt`  | `9_step_INVESTMENT_response.txt`  |

**사용 흐름**

1. `python main.py generate input/rfp.pdf --manual` → RFP 파싱, `manual_req_res/run_YYYYMMDD_HHMMSS/1_step_RFPAnalyzer_request.txt` 생성, `state.json` 초기화
2. **방법 A(자동화)**: `playwright install chromium` 후 `python main.py manual-step --site gemini`(또는 `--site chatgpt`) → 브라우저에서 로그인한 뒤 **같은 터미널에서 Enter** → 현재 단계 자동 전송·응답 저장  
   **방법 B(수동)**: 해당 request 파일 열기 → [시스템 프롬프트]·[사용자 메시지]를 Gemini/ChatGPT에 입력 → 응답을 해당 `*_response.txt`에 붙여넣기
3. `python main.py continue` → Step 1 응답 파싱, Step 2 요청 파일 생성
4. Step 2~9: `manual-step` 또는 수동으로 응답 저장 → `python main.py continue` 반복
5. Step 9 처리 후 `continue` → 모든 데이터 수집 완료 → `output/`에 PPTX 자동 생성  

**요청 파일 형식** (`N_step_{Phase명}_request.txt`)

요청 파일 안에는 [사용법], [시스템 프롬프트], [사용자 메시지]가 포함됩니다. 응답은 동일한 단계의 `N_step_{Phase명}_response.txt`에 붙여넣은 뒤 `python main.py continue` 실행.

**상태 파일** (`manual_req_res/run_YYYYMMDD_HHMMSS/state.json`)

진행 상태(version, rfp_path, project_name, client_name, current_step, total_steps, rfp_analysis, phase_contents, win_themes, cross_phase_summaries, started_at 등)를 시스템이 자동 관리합니다.

## 6. 신기능 상세

### RFP Chunking (긴 RFP 완전 분석)
- 한국어 공공 문서 헤딩 패턴 인식으로 섹션 분류
- 평가기준·배점·요구사항 섹션을 HIGH 우선순위로 전문 포함
- 기존 25,000자 제한 → 최대 40,000자로 확장

### Slide Quality Scoring (자동 채점)
- Phase 완료 후 규칙 기반으로 각 슬라이드 자동 채점 (0~100점)
- 항목: Action Title 준수(25%), 내용 풍부성(30%), 구체성(25%), 플레이스홀더(15%), 유형 적합성(5%)
- 평균 60점 미만 Phase는 경고 로그 출력

### Cross-Phase Context (내러티브 일관성)
- 이전 Phase의 핵심 결론(Win Theme, KPI, 전략 방향)을 다음 Phase 프롬프트에 자동 주입
- Phase 간 메시지 단절 방지

### Industry Stats DB (수치 구체성)
- 업종별 검증 통계를 프롬프트에 자동 삽입
- 예: marketing_pr → SNS 활용률, 콘텐츠 마케팅 ROI 등
- `INDUSTRY_STATS_PATH` 환경변수로 커스텀 통계 JSON 지정 가능

### Phase Checkpoint (안정성)
- 각 Phase 완료 후 `output/_checkpoints/run_YYYYMMDD_HHMMSS/` 폴더에 `phase_N_*.json` 저장
- Phase 1 완료 시 같은 폴더에 `run_metadata.json` 저장 (RFP 분석·Win Theme·프로젝트명 등 — 재개 시 사용)
- API 실패 시 **체크포인트 재개**로 이어서 진행 가능 (아래 참고)

### 체크포인트 재개 (--resume-checkpoint)
- 실행 중 API 오류 등으로 중단된 경우, 이미 저장된 Phase는 재사용하고 **부족한 Phase만** API로 생성한 뒤 PPTX까지 진행
- **인자**: `--resume-checkpoint`(또는 `-r`) 에 `_checkpoints/run_YYYYMMDD_HHMMSS` 또는 `run_YYYYMMDD_HHMMSS` 형식으로 지정

| 상황 | 동작 |
|------|------|
| Phase 1~7 전부 있음 | API 호출 없이 로드 후 바로 PPTX 생성 |
| Phase 1~4만 있음 | 1~4 로드, 5~7만 API로 생성 후 PPTX |
| run_metadata.json 없음 (구버전 run) | RFP 경로를 반드시 지정: `python main.py generate input/rfp.pdf --resume-checkpoint run_YYYYMMDD_HHMMSS` |

**예시:**
```bash
# 재개만 (run_metadata.json 있으면 RFP 경로 생략 가능)
python main.py generate --resume-checkpoint run_20260305_095658

# run_metadata 없을 때 RFP 지정
python main.py generate input/rfp.pdf --resume-checkpoint run_20260305_095658 -n "프로젝트명" -c "발주처"
```

## 8. 트러블슈팅

### API 키 오류

```
ANTHROPIC_API_KEY / GEMINI_API_KEY / GROQ_API_KEY가 설정되지 않았습니다.
```

`.env`에서 `LLM_PROVIDER`와 선택한 프로바이더에 맞는 키 확인. **Ollama** 사용 시에는 API 키가 필요 없습니다.

### Ollama 관련

| 현상                            | 확인·조치                                                                 |
| ------------------------------- | ------------------------------------------------------------------------- |
| **Connection refused / 연결 실패** | Ollama 실행 여부 확인. Windows/macOS: 앱 실행, Linux: `ollama serve`      |
| **model not found**             | `OLLAMA_MODEL`을 `ollama list` 이름과 맞추기. `ollama pull <모델명>` 후 `.env` 설정 |
| **응답이 느림**                  | 모델 크기·PC 사양(GPU) 영향. 작은 모델(예: llama3.2) 또는 GPU 드라이버 확인 |
| **JSON 파싱 실패·형식 깨짐**     | `LLM_TEMPERATURE=0.3`으로 낮추거나, qwen2.5:7b·llama3.1:8b 등 큰 모델 사용   |

### PDF 파싱 오류

- 텍스트 기반 PDF인지 확인 (스캔 이미지 PDF 미지원)
- 암호화된 PDF 미지원

### PPTX 생성 오류

```bash
pip install --upgrade python-pptx
```

### 슬라이드 품질 경고 로그

```
WARNING | Phase 3 품질 점수: 45.2점 (기준: 60점 미만)
```

프롬프트를 강화하거나 `.env`에서 `MIN_QUALITY_SCORE=40` 으로 기준 완화 가능.

### 429 할당량 초과 (API 사용 시)

- `LLM_PROVIDER=groq` 로 변경(무료 한도 넉넉) 또는 `LLM_PROVIDER=ollama` 로 로컬 LLM 사용.
- `.env`에 `LLM_DELAY_SECONDS=10` 등으로 호출 간 대기 늘리기.

### Phase 생성 실패 후 재시작

- `ENABLE_CHECKPOINT=true`(기본값)이면 `output/_checkpoints/run_YYYYMMDD_HHMMSS/`에 성공한 Phase와 `run_metadata.json`이 저장됨.
- **체크포인트 재개**: `python main.py generate --resume-checkpoint run_YYYYMMDD_HHMMSS` 로 해당 run 폴더를 지정하면, 저장된 Phase는 재사용하고 부족한 Phase만 API로 생성한 뒤 PPTX까지 진행. (run_metadata.json이 없으면 RFP 경로를 인자로 지정해야 함.)

### 폰트 표시 문제

- 템플릿에 Pretendard 등 미설치 폰트가 있으면 **맑은 고딕으로 자동 폴백**됨. 해당 폰트를 시스템에 설치하면 자동 적용.
